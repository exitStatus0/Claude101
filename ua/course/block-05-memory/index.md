## Проблема "нового працівника"

Ви бували в цій ситуації. Новий розробник приєднується до команди. Розумний, здібний — але перші два тижні кожен pull request повертається з одними й тими самими коментарями: "Ми тут не використовуємо default exports." "Наші K8s-лейбли дотримуються такої конвенції." "Тести йдуть у `__tests__/`, а не `test/`."

У Claude Code та сама проблема. З коробки він — блискучий генераліст, який нічого не знає про конвенції *вашого* проєкту. Кожна сесія починається з нуля, і ви повторюєте одне й те саме.

Пам'ять це виправляє. Назавжди.

## Що ми розглянемо

1. **Ієрархія пам'яті** — три рівні, що складаються як CSS-специфічність: користувач, проєкт, локальна
2. **Користувацький CLAUDE.md** — ваші персональні вподобання, що слідують за вами скрізь
3. **Проєктний CLAUDE.md** — стандарти команди, закомічені у версійний контроль (ви вже це бачили)
4. **CLAUDE.local.md** — специфіка вашої машини, секрети та шляхи, безпечно додані в gitignore
5. **Умовні правила** — файли `.claude/rules/`, що активуються лише при роботі з відповідними файлами
6. **Автопам'ять** — як Claude вчиться з ваших корекцій без написання жодного конфіг-рядка
7. **Команда `/memory`** — рентгенівський зір на те, що Claude знає про ваш проєкт

## Чому цей блок важливий

Ось у чому справа з AI-інструментами: різниця між "іноді корисний" та "незамінний" майже ніколи не в інтелекті моделі. Вона в контексті. Лікар, що не знає вашу історію хвороби, все одно може дати загальну пораду. Лікар із вашою повною картою? Це зовсім інша розмова.

Після цього блоку Claude Code знатиме конвенції комітів вашої команди, ваші патерни тестування, стратегію лейблів K8s і ваші персональні вподобання кодування. Кожну сесію. Автоматично. Без єдиного слова від вас.

## Передумови

- Завершені Блоки 0-4 (Claude Code встановлений, ai-coderrank працює локально, темна тема реалізована)
- Проєкт ai-coderrank відкритий у терміналі
- Готовність сказати Claude, як ви насправді любите працювати (будьте чесними — він не засуджуватиме)
